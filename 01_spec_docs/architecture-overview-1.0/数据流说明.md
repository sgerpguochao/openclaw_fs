## 二、典型数据流示例

### 示例 1：语音助手场景

```
1. 用户输入（macOS 客户端）
   用户说："Hey OpenClaw, 帮我看看这个错误"
   ↓
   macOS 客户端：
   - 语音识别转文字
   - 截取当前屏幕
   - 封装为 Gateway Protocol 消息

2. 发送到 Gateway（WebSocket）
   {
     "method": "agent.invoke",
     "params": {
       "message": "帮我看看这个错误",
       "attachments": [
         { "type": "image", "data": "base64..." }
       ]
     }
   }

3. Gateway 处理
   - 路由解析：识别用户身份和会话
   - 加载会话历史
   - 选择模型：Claude Opus 4.6
   - 准备工具：Bash, Read, Write 等

4. Agent 执行
   - 调用 Claude API
   - Claude 分析屏幕截图
   - Claude 返回："这是一个 TypeScript 类型错误..."

5. Gateway 分发结果
   - 格式化响应
   - 通过 WebSocket 发送回 macOS 客户端

6. macOS 客户端显示
   - 语音播报："这是一个 TypeScript 类型错误..."
   - 在界面显示详细解释
```

### 示例 2：跨渠道协作场景

```
1. 用户在 Telegram 发消息
   "帮我拍一张我桌面的照片"
   ↓
   Telegram Bot 接收消息

2. 发送到 Gateway
   Gateway 识别：
   - 用户：@username
   - 渠道：Telegram
   - 需要设备能力：Camera

3. Gateway 查找配对设备
   发现用户有配对的 iOS 设备（在线）

4. Gateway 请求 iOS 客户端
   通过 WebSocket 发送：
   {
     "method": "device.camera.capture",
     "params": { "reason": "User requested via Telegram" }
   }

5. iOS 客户端处理
   - 弹出权限确认："Telegram 会话请求拍照"
   - 用户批准
   - 调用摄像头拍照
   - 上传照片到 Gateway

6. Gateway 调用 Agent
   - 将照片和原始问题发送给 Claude
   - Claude 分析照片内容

7. Gateway 返回结果到 Telegram
   Telegram Bot 发送消息：
   "照片显示你的桌面上有一台 MacBook Pro..."
```

### 示例 3：命令执行审批场景

```
1. 用户在 Slack 发消息
   "帮我清理 /tmp 目录"
   ↓
   Slack Bot 接收

2. Gateway 路由到 Agent
   Agent (Claude) 生成工具调用：
   {
     "tool": "Bash",
     "command": "rm -rf /tmp/*"
   }

3. Gateway 检测到危险命令
   - 识别为需要审批的命令
   - 查找用户的配对设备
   - 发现 macOS 客户端在线

4. 发送审批请求到 macOS 客户端
   macOS 弹出审批窗口：
   ┌─────────────────────────────────┐
   │ 命令执行审批                     │
   ├─────────────────────────────────┤
   │ 来源：Slack (#general)          │
   │ 命令：rm -rf /tmp/*             │
   │                                 │
   │ ⚠️  警告：此命令将删除文件      │
   │                                 │
   │  [拒绝]  [修改]  [批准]         │
   └─────────────────────────────────┘

5. 用户批准
   macOS 客户端发送批准消息到 Gateway

6. Gateway 执行命令
   在沙箱环境中执行 rm -rf /tmp/*

7. 返回结果到 Slack
   Slack Bot 发送：
   "已清理 /tmp 目录，删除了 42 个文件"
```

## 三、关键技术点

### 1. 统一协议（Gateway Protocol）

所有客户端和渠道都使用统一的 Gateway Protocol 通信：

```typescript
// 消息格式
{
  "method": "agent.invoke" | "device.camera.capture" | "exec.approve",
  "params": { ... },
  "id": "request-id"
}

// 响应格式
{
  "result": { ... },
  "error": null,
  "id": "request-id"
}
```

### 2. 双向通信

- **客户端 → Gateway**：用户输入、设备能力提供
- **Gateway → 客户端**：结果返回、设备能力请求、审批请求

### 3. 多端同步

同一个会话可以在多个端点同时打开：
- 用户在 macOS 上开始对话
- 在 iOS 上继续对话
- 在 Telegram 上也能看到历史记录

### 4. 设备能力注册

客户端连接到 Gateway 时会注册自己的能力：

```json
{
  "device_id": "macos-macbook-pro",
  "capabilities": [
    "camera",
    "screen_capture",
    "bash_exec",
    "file_access",
    "voice_output"
  ]
}
```

Gateway 根据能力列表决定可以请求哪些操作。

## 四、数据流的关键特性

### 1. 渠道无关性

无论用户从哪个渠道发起请求（Telegram/Slack/macOS/iOS），Gateway 都会：
- 统一处理
- 统一路由
- 统一调用 Agent
- 根据渠道特性返回适配的结果

### 2. 设备能力按需调用

Agent 可以根据需要请求设备能力：
```
Agent: "我需要看到用户的屏幕"
Gateway: 查找有 screen_capture 能力的设备
Gateway → macOS 客户端: "请截图"
macOS 客户端 → Gateway: 返回截图
Gateway → Agent: 这是截图
```

### 3. 安全审批机制

敏感操作（命令执行、文件删除等）会触发审批流程：
- Gateway 检测到敏感操作
- 暂停 Agent 执行
- 请求用户审批
- 用户批准后继续执行

### 4. 流式响应

Agent 的输出可以实时流式传输：
```
Agent 开始生成 → Gateway 实时转发 → 客户端实时显示
```

用户不需要等待完整响应，可以看到 AI 的"思考过程"。

## 五、总结

你的理解完全正确！核心要点：

1. **客户端提供能力**：摄像头、屏幕、位置、语音等
2. **Gateway 统一控制**：路由、会话、安全、分发
3. **Agent 执行指令**：调用 AI 模型、执行工具
4. **结果返回客户端**：显示、播报、渲染

**Gateway 是整个系统的"大脑"**，所有数据流都经过它：
- 接收来自多个渠道的输入
- 统一调度和处理
- 按需请求设备能力
- 分发结果到各个端点

这种架构的优势：
- **统一管理**：一个 Gateway 管理所有渠道和设备
- **灵活扩展**：新增渠道或设备只需实现 Gateway Protocol
- **安全可控**：所有敏感操作都经过 Gateway 审批
- **多端协作**：不同设备可以协同工作

# OpenClaw 系统架构说明 v1.0

## 1. 文档目标与范围

本文基于当前仓库代码（`e:\openclaw`）重新梳理 OpenClaw 最新系统架构，覆盖：

- 核心运行时分层与模块职责
- Gateway 控制平面与 Agent 执行平面的协同
- Channels / Plugins / Agent 的最新关系模型
- 客户端（CLI / Web / macOS / iOS / Android）接入架构
- 相比旧版《系统架构说明.md》的改动与新增能力

不覆盖：平台部署细节和具体函数级实现。

---

## 2. 最新总体定位

OpenClaw 已从“多渠道 AI 网关”演进为“三平面协同”架构：

1. **Control Plane（Gateway）**：统一 WS/HTTP 协议面，负责鉴权、路由、编排、状态分发。
2. **Execution Plane（Agent Runtime）**：负责模型调用、工具执行、会话管理、技能与沙箱策略。
3. **Device/Node Plane（Node Host）**：承载设备能力（camera/screen/location/system.run 等），通过 `node.invoke` 纳入统一调度。

核心入口仍是：

- CLI 启动：`openclaw.mjs` -> `src/entry.ts` -> `src/cli/run-main.ts`
- Gateway 启动：`gateway run` -> `src/gateway/server.ts` -> `src/gateway/server.impl.ts`

## 3. 顶层模块分层（最新）

## 3.1 入口与 CLI 编排层

- `openclaw.mjs`：Node 可执行入口 + dist 加载。
- `src/entry.ts`：主入口守卫、环境初始化、CLI 重启策略（ExperimentalWarning 抑制）。
- `src/cli/`：命令注册与子 CLI 懒加载。
- `src/commands/`：命令业务实现（agent/message/gateway/status/onboard 等）。

当前子 CLI 体系已显著扩展，包含：
`gateway`、`daemon`、`models`、`nodes`、`devices`、`node`、`sandbox`、`cron`、`hooks`、`webhooks`、`plugins`、`channels`、`pairing`、`security`、`skills`、`update`、`docs`、`tui` 等。

## 3.2 Gateway 控制平面

- 核心装配：`src/gateway/server.impl.ts`
- HTTP 面：`src/gateway/server-http.ts`
- WS 协议与方法分发：`src/gateway/server-methods.ts` + `src/gateway/server-methods/*`
- 协议模型：`src/gateway/protocol/schema/*`

Gateway 现在是“多子系统装配器”，在启动中统一挂载：

- 插件注册表
- Channel manager
- 节点注册表（NodeRegistry）
- Cron 服务
- 鉴权与限流
- 配置热重载
- Tailscale 暴露
- Browser/Canvas 等 sidecar

## 3.3 Agent 执行平面

- 核心运行链：`src/agents/pi-embedded-runner.ts` 及 `src/agents/pi-embedded-runner/*`
- 工具体系：`src/agents/tools/*` + `src/agents/openclaw-tools.ts`
- 策略体系：`tool-policy*`、`transcript-policy*`、`context-window-guard*`
- 会话与子代理：`subagent-registry*`、`sessions-*`
- 技能系统：`src/agents/skills*`
- 沙箱系统：`src/agents/sandbox/*`

## 3.4 渠道接入与统一适配层

- 通用渠道抽象：`src/channels/*`
- 渠道插件协议：`src/channels/plugins/types.*`
- 渠道插件运行时：`src/channels/plugins/index.ts`
- 路由：`src/routing/resolve-route.ts`

路由匹配优先级已细化为：
`peer` -> `parent peer` -> `guild+roles` -> `guild` -> `team` -> `account` -> `channel` -> `default`。

## 3.5 插件扩展层

- 插件发现/加载：`src/plugins/discovery.ts`、`src/plugins/loader.ts`
- 插件注册表：`src/plugins/registry.ts`
- 运行态：`src/plugins/runtime.ts`

插件能力面已覆盖：

- `tools`
- `channels`
- `gateway methods`
- `http handlers/routes`
- `hooks`（含 typed hooks）
- `cli`
- `providers`
- `services`
- `commands`

## 3.6 设备节点与侧车能力

- Node Host：`src/node-host/*`
- Browser 控制服务：`src/browser/*`
- Canvas Host：`src/canvas-host/*`

这部分使 OpenClaw 不再仅是消息机器人，而是可把“本机/移动设备能力”纳入统一调用面。

## 3.7 客户端与 UI 层

- Web UI：`ui/`
- macOS：`apps/macos/`
- iOS：`apps/ios/`
- Android：`apps/android/`
- 共享 SDK：`apps/shared/OpenClawKit/`

## 4. 关键运行链路

## 4.1 消息入站到 Agent

`Channel Adapter` -> `resolveAgentRoute()` -> `sessionKey` 生成 -> `agent` 执行 -> `Channel Outbound` 回发。

关键点：

- 会话键不再只按 channel 简单映射，已支持 `dmScope`、`identityLinks` 等策略。
- 路由支持 Discord 角色、群组上下文、线程父会话继承。

## 4.2 Gateway WS 控制链路

客户端建立 WS 后，必须先发 `connect`，通过鉴权/作用域校验后才能调用方法。

当前基础方法面（节选，见 `server-methods-list.ts`）：

- 系统：`health`、`status`、`usage.*`、`logs.tail`
- 配置：`config.get/set/apply/patch/schema`
- Agent：`agent`、`agent.wait`、`agents.*`
- 会话：`sessions.*`
- 技能：`skills.*`
- 渠道：`channels.status/logout`、`send`
- 自动化：`cron.*`、`wizard.*`
- 节点：`node.*`、`device.*`
- 浏览器：`browser.request`
- WebChat：`chat.send/history/abort`

## 4.3 Node 能力调用链路

`operator/agent` -> `node.invoke` -> `node` 会话执行（iOS/Android/macOS/node-host）-> `node.invoke.result` + `node.event`。

配套能力：

- 设备配对审批（`node.pair.*`, `device.pair.*`）
- 节点 token 轮换（`device.token.*`）
- 命令策略与审批（exec approvals）

## 4.4 Gateway 热重载与维护链路

- 配置监听：`startGatewayConfigReloader`
- 重载策略：`off/restart/hot/hybrid`
- 运行期任务：heartbeat、health monitor、cron、delivery recovery、update check

## 5. Channels / Plugins / Agent 最新关系

## 5.1 核心关系

- **Plugin**：能力注册容器。
- **Channel**：一种特殊 plugin 能力（通过 `registerChannel` 注入 registry）。
- **Agent**：消费工具与策略的执行单元，不直接耦合具体渠道 SDK。

## 5.2 运行时关系

1. 启动时，插件系统构建 `PluginRegistry`。
2. Channel 插件从 registry 暴露给 Gateway。
3. 入站消息经 routing 绑定到 agent。
4. Agent 执行时按策略消费 registry 中工具/钩子/服务。

## 5.3 当前渠道生态

- 核心内置目录：`src/telegram`、`src/slack`、`src/discord`、`src/signal`、`src/imessage`、`src/whatsapp`、`src/web`、`src/line`
- 扩展渠道（`extensions/*`）：`msteams`、`matrix`、`zalo`、`zalouser`、`googlechat`、`feishu`、`mattermost`、`nextcloud-talk`、`twitch`、`irc`、`nostr`、`synology-chat` 等

## 6. 安全与策略架构（新增重点）

相比旧版，“安全边界”已成为架构一级能力：

1. **连接鉴权模式扩展**：`none/token/password/trusted-proxy`。
2. **鉴权限流**：auth rate limit（窗口、锁定时长、loopback 例外）。
3. **Control UI 安全开关**：origin 控制、device auth、insecure auth 控制。
4. **DM Pairing 与 Allowlist**：渠道默认安全接入策略。
5. **Exec Approval**：高风险命令审批与转发。
6. **Gateway 工具面限制**：`gateway.tools.allow/deny`。
7. **Node 命令策略**：`gateway.nodes.allowCommands/denyCommands`。

---

## 7. 客户端接入架构（统一协议，多端实现）

所有客户端都收敛到同一 Gateway 协议面，但形态不同：

- CLI：管理与脚本编排优先
- Web UI：浏览器控制台
- macOS/iOS/Android：operator + node 双角色协同

共同点：

- 都通过 WS `connect` 握手进入 RPC
- 都受 role/scopes + auth 策略约束
- 都消费统一事件总线（chat/agent/presence/node/cron 等）

---

## 8. 与旧版文档相比的关键改动

## 8.1 架构层级变化

旧版主要是“CLI + Gateway + Agent + Channels + Plugins”。
新版应升级为：

- `Gateway Control Plane`
- `Agent Execution Plane`
- `Node/Device Plane`
- `Browser/Canvas Sidecars`

## 8.2 能力面扩展

新增/强化能力包括：

1. **Node 体系**：节点注册、配对审批、设备 token 生命周期、`node.invoke`。
2. **Browser 服务**：独立浏览器控制与代理链路。
3. **Canvas Host**：A2UI 画布托管与联动。
4. **OpenResponses HTTP 面**：`/v1/responses` 及文件/图片安全策略。
5. **配置热重载体系**：`hot/restart/hybrid`。
6. **安全增强**：trusted-proxy、auth 限流、exec approvals、origin 与设备认证。
7. **路由策略增强**：parent peer、guild+roles、team 等维度。
8. **插件注册能力增强**：typed hooks、services、http route、provider 等。
9. **CLI 运营化**：onboard、skills、security、devices、nodes、update、docs、tui。

## 8.3 渠道生态变化

旧版文档只覆盖 Telegram/Slack/Discord/Signal/iMessage/WhatsApp。
新版应明确：

- 渠道是 plugin 化加载，来源可为 core + extensions。
- 渠道能力由统一 adapter 协议定义（config/gateway/outbound/status/messaging 等）。

## 10. 一句话总结

OpenClaw 最新架构的本质是：

**以 Gateway 作为统一控制平面，把多渠道消息、Agent 执行、设备节点能力、浏览器/画布侧车能力全部收敛到同一协议与策略系统中，并通过插件机制持续扩展。**
